# üìù Testing Documents & Related Terms

---

## 1. **Test Plan**

* A **formal document** describing the **scope, objectives, strategy, resources, schedule, and deliverables** of testing.
* It answers **what to test, how to test, who will test, and when testing will be done**.

---

## 2. **Test Scenario**

* A **high-level description of what to test** ‚Üí usually derived from requirements.
* Example: ‚ÄúVerify login functionality with valid and invalid credentials.‚Äù

---

## 3. **Test Case**

* A set of **step-by-step instructions with input, execution steps, and expected result** to validate a feature.
* Example: Steps to enter username & password ‚Üí Expected: User should log in successfully.

---

## 4. **Test Script**

* A set of **automated test instructions** written using tools (e.g., Selenium + Java).
* It performs **execution of test cases automatically**.

---

## 5. **Test Data**

* The **input values (valid, invalid, boundary, large data sets)** used to test application behavior.
* Example: Test login using valid email, invalid email, empty field, etc.

---

## 6. **Test Execution**

* The process of **running test cases or test scripts** and recording the results.
* Can be **manual (step execution)** or **automation (tool execution)**.

---

## 7. **Test Report**

* A **summary document of test execution results** showing total test cases, passed, failed, blocked, skipped.
* Helps stakeholders **understand project quality and risks**.

---

## 8. **Defect / Bug**

* A **mismatch between expected result and actual result** found during testing.
* Example: Clicking login with valid credentials ‚Üí Actual: Error shown.

---

## 9. **Defect Life Cycle**

* The **journey of a defect from identification to closure**.
* States: New ‚Üí Assigned ‚Üí Open ‚Üí Fixed ‚Üí Retested ‚Üí Verified ‚Üí Closed (or Reopened if not fixed).

---

## 10. **RTM (Requirement Traceability Matrix)**

* A document that **maps requirements to test cases** to ensure full coverage.
* Helps confirm that **every requirement has been tested**.

---

## 11. **Smoke Testing**

* A quick test to check whether the **basic functionalities of the build are working**.
* Example: Check login, signup, and homepage load before deeper testing.

---

## 12. **Sanity Testing**

* Narrow and deep testing to check whether **a specific bug fix or module is working properly**.
* Example: After fixing login issue ‚Üí only test login page in detail.

---

## 13. **Regression Testing**

* Testing to ensure that **new changes do not break existing features**.
* Example: After adding payment option ‚Üí recheck login, cart, checkout.

---

## 14. **Re-testing**

* Running the **same test again** after fixing a defect.
* Example: Login was failing ‚Üí after fix ‚Üí retest login.

---

## 15. **Test Strategy**

* A **high-level document** prepared by the QA Manager that defines **approach, tools, and testing levels**.
* Test Plan = project-specific; Test Strategy = organization-level.

---

## 16. **Test Environment**

* The **hardware, software, database, network setup** required for testing.
* Example: Windows 10, Chrome browser, MySQL DB, Test server.

---

## 17. **Entry & Exit Criteria**

* **Entry Criteria** ‚Üí Conditions that must be met before testing begins (e.g., test environment ready).
* **Exit Criteria** ‚Üí Conditions that define when testing is completed (e.g., 95% test cases passed).

---

## 18. **Bug Severity & Priority**

* **Severity** ‚Üí Impact of defect on system (Critical, Major, Minor).
* **Priority** ‚Üí Order in which bug should be fixed (High, Medium, Low).
* Example: Spelling mistake (Low severity, Low priority) vs. Login crash (High severity, High priority).

---

## 19. **Levels of Testing**

* **Unit Testing** ‚Üí Individual module testing.
* **Integration Testing** ‚Üí Module interaction testing.
* **System Testing** ‚Üí Entire application testing.
* **UAT (User Acceptance Testing)** ‚Üí Testing by end users/business.

---

## 20. **Types of Testing (Common in Interviews)**

* **Functional Testing** ‚Üí Check business functionalities.
* **Non-Functional Testing** ‚Üí Performance, security, usability.
* **Black Box Testing** ‚Üí Tester doesn‚Äôt see code, only inputs/outputs.
* **White Box Testing** ‚Üí Tester checks code structure, logic, loops.
* **Grey Box Testing** ‚Üí Partial code knowledge.

# üìò Complete Java Notes (Detailed + Beginner Friendly)

---

## 1. Introduction to Java

* Java is a **high-level, object-oriented, secure, and platform-independent language** developed by Sun Microsystems (1995).
* It compiles code into **bytecode** which runs on the **JVM (Java Virtual Machine)**.
* Known for *portability* ‚Üí *‚ÄúWrite Once, Run Anywhere‚Äù*.

---

## 2. Java Architecture (JDK, JRE, JVM)

* **JVM (Java Virtual Machine)** ‚Üí executes bytecode, makes Java platform-independent.
* **JRE (Java Runtime Environment)** ‚Üí JVM + libraries to run programs.
* **JDK (Java Development Kit)** ‚Üí JRE + tools like compiler, debugger (for development).

---

## 3. Data Types in Java

* **Primitive (8 types):**

  * byte (1 byte), short (2), int (4), long (8) ‚Üí integers.
  * float (4), double (8) ‚Üí decimals.
  * char (2) ‚Üí single character.
  * boolean (1 bit) ‚Üí true/false.
* **Non-Primitive:** Strings, Arrays, Classes, Interfaces, Objects.

---

## 4. Variables

* **Local Variables** ‚Üí declared inside methods, exist during method execution.
* **Instance Variables** ‚Üí object-specific, stored in heap memory.
* **Static Variables** ‚Üí shared across all objects, memory-efficient.
* **Final Variables** ‚Üí constants, cannot change after initialization.

---

## 5. Operators

* **Arithmetic** ‚Üí `+ - * / %`
* **Relational** ‚Üí `== != > < <= >=`
* **Logical** ‚Üí `&& || !`
* **Assignment** ‚Üí `= += -= *= /=`
* **Unary** ‚Üí `++ --` (increment/decrement)
* **Ternary** ‚Üí `(condition) ? trueValue : falseValue`

---

## 6. Control Statements

* **if-else** ‚Üí decision making.
* **switch-case** ‚Üí multiple options based on a value.
* **Loops:**

  * for ‚Üí known iterations.
  * while ‚Üí condition-based loop.
  * do-while ‚Üí runs at least once.
* **break** ‚Üí exits loop, **continue** ‚Üí skips current iteration.

---

## 7. OOPs Concepts in Java

Java follows **4 main pillars of OOP**:

1. **Encapsulation**

   * Binding data and methods into a single unit (class).
   * Achieved using **private variables + public getters/setters**.

2. **Inheritance**

   * Acquiring properties of one class into another (`extends`).
   * Promotes **code reusability**.
   * Types: single, multilevel, hierarchical (Java doesn‚Äôt support multiple inheritance directly).

3. **Polymorphism**

   * **Compile-time (Overloading):** Same method name, different parameters.
   * **Runtime (Overriding):** Subclass provides specific implementation of parent class method.

4. **Abstraction**

   * Hiding internal details, showing only essential features.
   * Achieved using **abstract classes** (incomplete methods) and **interfaces**.

---

## 8. Important Keywords

* **this** ‚Üí refers to current object.
* **super** ‚Üí calls parent class constructor/method.
* **final** ‚Üí makes constants, prevents inheritance or overriding.
* **static** ‚Üí class-level keyword (variables, methods).
* **abstract** ‚Üí defines incomplete methods/classes.
* **synchronized** ‚Üí ensures thread safety.
* **volatile** ‚Üí variable value updated by multiple threads.
* **transient** ‚Üí skips variable during serialization.

---

## 9. Strings

* **Immutable:** Once created, values cannot change.
* Declared as `"Hello"` or `new String("Hello")`.
* **Important Methods:**

  * `length()`, `charAt(i)`, `substring()`, `equals()`, `compareTo()`, `concat()`, `toLowerCase()`, `toUpperCase()`.
* For **mutable strings**, Java provides **StringBuilder & StringBuffer**.

---

## 10. Arrays

* **Definition:** Fixed-size collection of same type elements stored in sequence.
* Example: `int arr[] = new int[5];`
* Types:

  * Single Dimensional (`int arr[]`).
  * Multi-Dimensional (`int arr[][]`).
* Traversed using normal `for` or `for-each` loop.

---

## 11. Exception Handling

* **Definition:** Mechanism to handle runtime errors gracefully.
* **Keywords:**

  * `try` ‚Üí code to test.
  * `catch` ‚Üí handle exception.
  * `finally` ‚Üí always executes (cleanup).
  * `throw` ‚Üí manually throw exception.
  * `throws` ‚Üí declare exceptions in method signature.
* **Types:**

  * Checked (compile-time) ‚Üí e.g., IOException.
  * Unchecked (runtime) ‚Üí e.g., NullPointerException.

---

## 12. Collections Framework

* Provides **ready-made data structures** for storing/managing data.
* **List** (ordered, allows duplicates): `ArrayList`, `LinkedList`.
* **Set** (unique elements, no duplicates): `HashSet`, `TreeSet`.
* **Map** (key-value pairs): `HashMap`, `TreeMap`.
* **Queue** (FIFO): `PriorityQueue`, `Deque`.

---

## 13. Multithreading

* **Definition:** Executing multiple tasks simultaneously.
* **Creating Threads:**

  1. Extending `Thread` class and overriding `run()`.
  2. Implementing `Runnable` interface.
* **Important Methods:** `start()`, `run()`, `sleep()`, `join()`.
* **Synchronization:** prevents data inconsistency when multiple threads access shared resources.

---

## 14. File Handling (I/O in Java)

* Java provides **java.io** package for file operations.
* **Classes:**

  * `File` ‚Üí represents file.
  * `FileReader`, `FileWriter` ‚Üí read/write.
  * `BufferedReader`, `BufferedWriter` ‚Üí efficient reading/writing.
* **try-with-resources** ‚Üí auto-closes files after use.

---

## 15. Java 8 Features

* **Lambda Expressions:** `(a, b) -> a + b`, shorter anonymous methods.
* **Functional Interfaces:** Interfaces with a single abstract method (`@FunctionalInterface`).
* **Streams API:** Process collections (filter, map, reduce) in functional style.
* **Default Methods:** Allow methods with body in interfaces.
* **Date & Time API (java.time):** Improved date handling.

---

## 16. Memory Management

* **Stack Memory:** Stores method calls and local variables.
* **Heap Memory:** Stores objects and instance variables.
* **Garbage Collector (GC):** Automatically deletes unused objects.
* **Reference types:** Strong, weak, soft, phantom references.

---

## 17. Access Modifiers

* **public** ‚Üí accessible everywhere.
* **protected** ‚Üí accessible in same package + subclasses.
* **default (no keyword)** ‚Üí accessible only in same package.
* **private** ‚Üí accessible only within the class.

---

## 18. Differences (Common Interview Qs)

* **JDK vs JRE vs JVM:** JDK = development tools, JRE = runtime, JVM = executor.
* **Overloading vs Overriding:** Overloading ‚Üí same method name, diff parameters (compile-time). Overriding ‚Üí same method signature, redefined in child (runtime).
* **Abstract Class vs Interface:** Abstract class can have abstract + non-abstract methods; interface has only abstract methods (till Java 7, Java 8 added default methods).
* **String vs StringBuilder vs StringBuffer:** String ‚Üí immutable, StringBuilder ‚Üí mutable, not thread-safe, StringBuffer ‚Üí mutable, thread-safe.

### üîπ Based on Level

1. **Unit Testing** ‚Üí testing individual components (done by developers).
2. **Integration Testing** ‚Üí checking data flow between modules.

### üîπ Based on Approach

* **Functional Testing** ‚Üí validates features (login, signup, cart, etc.).

## 11. Agile Testing

# ü§ñ Automation Testing + Selenium WebDriver Notes

---

## 1. What is Automation Testing?

* Automation Testing is the process of **executing test cases using tools or scripts** instead of manual execution.
* Purpose ‚Üí **save time, increase coverage, reduce human error, and improve accuracy**.

---

## 2. When to Use Automation Testing?

* Repetitive test cases (like regression testing).
* Large test data execution.
* Cross-browser/cross-platform testing.
* Load/performance testing.
* Not suitable for: one-time or exploratory testing.

---

## 3. Popular Automation Testing Tools

* **Selenium (Java/Python, open-source)**
* **TestNG / JUnit** ‚Üí test frameworks
* **Cucumber (BDD)**
* **Cypress (JS-based)**
* **Playwright, Puppeteer**
* **Performance tools** ‚Üí JMeter, LoadRunner

---

## 4. Selenium Overview

* Selenium is an **open-source automation tool for web applications**.
* Supports multiple languages (Java, Python, C#, JavaScript).
* Supports all major browsers (Chrome, Firefox, Edge, Safari).
* **Components:**

  1. **Selenium IDE** ‚Üí record & playback (basic).
  2. **Selenium RC (deprecated)**.
  3. **Selenium WebDriver** ‚Üí advanced, real browser interaction.
  4. **Selenium Grid** ‚Üí run tests on multiple machines/browsers in parallel.

---

## 5. Selenium WebDriver

* **Definition:** WebDriver is a library that directly communicates with browsers using browser-specific drivers.
* Provides **API methods** to locate elements, interact with them, navigate pages, handle alerts, etc.
* Supported Browsers ‚Üí Chrome, Firefox (GeckoDriver), Edge, Safari.

---

## 6. Locators in Selenium (Most Important)

Locators help Selenium find web elements.

* **By ID** ‚Üí `driver.findElement(By.id("username"))`
* **By Name** ‚Üí `driver.findElement(By.name("password"))`
* **By ClassName** ‚Üí `driver.findElement(By.className("btn"))`
* **By TagName** ‚Üí `driver.findElement(By.tagName("input"))`
* **By LinkText** ‚Üí `driver.findElement(By.linkText("Login"))`
* **By PartialLinkText** ‚Üí `driver.findElement(By.partialLinkText("Log"))`
* **By CSS Selector** ‚Üí `driver.findElement(By.cssSelector("input#email"))`
* **By XPath** ‚Üí `driver.findElement(By.xpath("//input[@id='email']"))`

## 13. Common Interview Q\&A

* **Difference: Selenium WebDriver vs Selenium RC**

* **Difference: findElement() vs findElements()**

  * `findElement()` ‚Üí returns first matching element.
  * `findElements()` ‚Üí returns list of all matching elements.

* **Implicit vs Explicit Wait**

  * Implicit ‚Üí applies globally, waits for element before throwing exception.
  * Explicit ‚Üí applies to specific element with condition.

  * TestNG supports parallel execution, flexible annotations, better reporting, priorities, and grouping.

  * A design pattern that separates locators and test logic ‚Üí improves maintainability.

# üß™ TestNG Notes

---

## 1. What is TestNG?

* **TestNG (Test Next Generation)** is a **testing framework inspired by JUnit and NUnit**.
* Provides advanced features like **annotations, parallel execution, grouping, prioritization, and reporting**.

---

## 2. Why TestNG in Selenium?

* Generates **HTML test reports automatically**.
* Supports **parallel execution** to save time.
* Allows **test case grouping and prioritization**.
* Handles **dependency testing** (run tests in sequence).
* Better suited than JUnit for **large automation projects**.

---

## 3. TestNG Annotations (Important)

* `@Test` ‚Üí Marks a method as a test case.
* `@BeforeMethod` ‚Üí Runs before each test case.
* `@AfterMethod` ‚Üí Runs after each test case.
* `@BeforeClass` ‚Üí Runs once before all tests in a class.
* `@AfterClass` ‚Üí Runs once after all tests in a class.
* `@BeforeTest` ‚Üí Runs before `<test>` tag in XML.
* `@AfterTest` ‚Üí Runs after `<test>` tag in XML.
* `@BeforeSuite` / `@AfterSuite` ‚Üí Runs before/after whole test suite.

---

## 4. TestNG Features

* **Prioritization** ‚Üí

  ```java
  @Test(priority=1)
  public void loginTest() {}
  @Test(priority=2)
  public void dashboardTest() {}
  ```
* **Grouping** ‚Üí

  ```java
  @Test(groups={"Smoke"})
  public void test1() {}
  @Test(groups={"Regression"})
  public void test2() {}
  ```
* **Dependency** ‚Üí

  ```java
  @Test
  public void startApp() {}

  @Test(dependsOnMethods={"startApp"})
  public void loginTest() {}
  ```
* **Parallel Execution** ‚Üí defined in `testng.xml`.

---

## 5. testng.xml File

* XML configuration file to control test execution.
* Example:

```xml
<suite name="AutomationSuite">
  <test name="RegressionTests" parallel="methods" thread-count="2">
    <classes>
      <class name="tests.LoginTest"/>
      <class name="tests.DashboardTest"/>
    </classes>
  </test>
</suite>
```

---

## 6. Reporting in TestNG

* Automatically generates **HTML and XML reports** after execution.
* Can be enhanced using **Extent Reports or Allure Reports**.

---

## 7. Advantages of TestNG

* Flexible annotations.
* Supports parallel execution.
* Built-in HTML reports.
* Easy integration with **Maven/Gradle + Jenkins (CI/CD)**.
